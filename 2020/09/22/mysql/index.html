mysql索引底层原理

insert的时候进行排序（原因：查远远大于写的次数）
每次innodb会取16kb（一页的数据出来），因为就算取一行，肯定也会取一页，放到内存里

页目录可以设定一些索引，通过顺序排序，二分查找能找到对应的数据

如果不用自增ID做主键，那么如果要保证顺序，那么要进行数据的交换。

主键索引=聚集索引，索引和数据在一起

联合索引存储主键索引，再从主键去找落盘的数据

为什么b>1不走索引，b>6走索引？

因为b>1，需要回表很多次，b>6只需要回表两次

hash存储不支持范围查找，等值查hash，能找到对应的链表，找到数据

索引的设计原则

buffer pool

磁盘与buffer pool的关系

free链表存储空白的buffer pool位置
flush链表存储update的位置，方便刷入磁盘
lru链表，最近访问的放链表头，那么最后的位置是冷数据位置，可以淘汰
5/8 热数据区域 3/8 冷数据区域 淘汰冷数据区域，并且新加入的数据加入冷数据区域 如果再次访问加载到热数据区域头部

log

buffer pool 刷新到磁盘的时候 mysql挂了

redolog

innodb自带的，记录的是哪一页哪个位置的数据的修改

方式一：
1.修改buffer pool里面的页数据—脏页
2.update语句 –> 生成一个redolog –> log buffer
3.redolog 持久化 事务提交去持久化
4.修改成功

方式二：
1.修改buffer pool里面的页数据—脏页
2.修改磁盘中的页数据
3.修改成功

为什么方式一快，因为对于某一页来说，用户数据区域可能在磁盘不同的位置，要去修改，那么磁盘要一直转，因为存储数据的时候不一定是连续存储。导致效率低，这也是机械硬盘速度慢的原因之一。

mysql有提前分配一块区域 redolog ib_logfile0 ib_logfile1 文件大小

如果redolog满了，那么要检查buffer pool，开始刷盘，然后清空redolog。如果redolog设置大一点，那么这种刷盘操作就减少了，提高了性能

binlog

mysql自带的，记录的是真的sql语句

binlog实际是除了select以外所有操作数据库动作的记录，方便恢复数据库。master主动推送到slave结点的relay log，从结点io线程去读relay log然后在sql线程去更新数据库。

undo log

1.修改buffer pool里面的页数据—脏页
2.update语句 –> 生成一个redolog –> log buffer
3.redolog 持久化 事务提交去持久化
4.binlog 持久化
5.undo log反向操作
6.修改成功

引擎

Myisam(表级锁) Innodb(行级锁)
Myisam存储了表的总行数，所以count特别快

nnoDB 支持崩溃后安全恢复，MyISAM 不支持崩溃后安全恢复；
InnoDB 支持行级锁，MyISAM 不支持行级锁，只支持到表锁；
InnoDB 支持外键，MyISAM 不支持外键；
MyISAM 性能比 InnoDB 高；
MyISAM 支持 FULLTEXT 类型的全文索引，InnoDB 不支持 FULLTEXT 类型的全文索引，但是 InnoDB 可以使用 sphinx 插件支持全文索引，并且效果更好；
InnoDB 主键查询性能高于 MyISAM。

1.为什么Myisam快？
因为Myisam B+树存的是引用，加载到内存快。 等Innodb加载完，Myisam都查完了
2.为什么Innodb不支持全文索引？
数据直接在叶子结点上，如果要全文索引，相当于复制了一份数据库，浪费空间

视图

获取数据更容易，相对于多表查询来说；
视图能够对机密数据提供安全保护；
视图的修改不会影响基本表，提供了独立的操作单元，比较轻量。

共享锁
读锁，S锁

排他锁
写锁，X锁

表锁

行锁

记录锁
记录锁也属于行锁的一种，只不过记录锁的范围只是表中的某一条记录。精准命中，并且命中的条件字段是唯一索引

页锁
介于行锁和表锁中间的一种锁。

间隙锁
行锁的一种，记录的是事务加锁后的某一个区间，遵循左开右闭原则。

临建锁
也属于行锁的一种，是INNODB的行锁的默认算法，是记录锁和间隙锁的组合，临建锁会把查询出来的记录锁住，同时也会把该范围内的所有间隙锁空间也会锁住，再之它会把相邻的下一个区间也会锁住

如果当事务A加锁成功后就设置一个状态告诉后面的人，已经有人对表里的行加了一个排他锁了，你们不能对整个表加共享锁或排他锁，那么后面需要对整个表加锁的人只需要获取这个状态就知道自己是不是可以对表加锁，避免了对整个索引树的每个节点扫描是否加锁，这个状态就是意向锁。

意向共享锁
当一个事务试图对整个表进行加共享锁之前，首先需要获得这个表的意向共享锁。

意向排他锁
当一个事务试图对整个表进行加排他锁之前，首先需要获得这个表的意向排他锁。

MVCC

多版本并发控制：读取数据时通过一种类似快照的方式将数据保存下来，这样读锁和写锁就不冲突了，不同的事务session会看到自己特定版本的数据

MVCC只在READ COMMITTED和REPEATABLE READ两个隔离级别下工作。其他两个隔离级别和MVCC不兼容，因为READ UNCOMMITTED总是读取最新的数据行，而不是符合当前事务版本的数据行。而SERIALIZABLE则会对所有读取的行都加锁。

聚簇索引记录中有两个必要的隐藏列：
trx_id：用来存储每次对某条聚簇索引记录进行修改的时候的事物ID。
roll_pointer：每次对哪条聚簇索引记录有修改的时候，都会把老版本写入undo日志中。这个roll_pointer就是存了一个指针，它指向这条聚簇索引记录的上一个版本的位置，通过它来获得上一个版本的记录信息。

已提交读和可重复读的区别就在它们生成ReadView的策略不同。

开始事务时创建ReadView(ReadView维护当前的事务ID，即未提交的事务ID，排序生成一个数组)

访问数据，获取数组的事务ID（获取最大的事务ID），对比ReadView：

如果在ReadView左边，代表已经提交了，可以访问；在右边就不可以，获取roll_pointer，取上一个版本重新对比

已提交读每次查询都会生成一个独立的ReadView，可重复读在第一次读的时候生成一个ReadView，复用之前的ReadView
